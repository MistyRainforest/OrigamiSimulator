Origami Simulator (Magnetic Force Add-on)

Basic Layout of Origami Simulator

Globals initializes model, solver and gpumath based on the initial parameters given, which includes angle of fold/stiffness of axis etc.

Implementing Magnetic Field

Magnetic field equations were imported from a gazebo plugin, and the defunct externalForce() function was repurposed to update magnetic field while solving the folds.

Challenge 1: Updating Force
The implementation of gpumath parallelized the integration of parameters to speed up the originally linear process. This meant that the equations within gpumath looked at the each node separately which made it difficult to unpack when trying to add a new force on top of the simulation.
Solution 1
We repurposed the updateExternalForce() function to be called at each solution step instead of upon a change in external force. This led to a drastic slowdown in processing speed which grew linearly with the number of nodes. The updateExternalForce() function was then further modified to only update at certain intervals instead of all steps when solving for the next displacement, which reduced the required steps required, scaling it down by the magnitude of interval chosen.

Challenge 2: Interactive Magnet
Interacting with a 3d environment with a 2d controls reduced the freedom of movement from the full 3 dimensions to an arbitrarily fixed 2 dimension plane, which might not always match up with the current model.

Solution 2
Main interactive magnet was shifted to reside in a node, which was placed as a variable in globals. The magnet could then be placed in the 3 dimensional space using the already implemented UI.

Challenge 3: Lack of Collision other Physics Fundamentals
The lack of collision, gravity and friction meant that after assigning magnetic forces to the nodes, they behaved as though suspended in space. Which led to the models not deforming but moving according to orbital mechanics.

Solution 3
An attempt at filling in the missing elements was made however the work required was unfeasible. The alternative was to enable certain points to be fixed in space to provide a counteracting force which enabled the model to be deformed in a manner which more closely resembled the end goal. This was achieved by re-purposing the user interaction function.

Challenge 4: Speedup + Magnetization of Individual Nodes
The implementation of updateExternalForce() was still much too slow at large numbers of nodes. Furthermore the magnetization of nodes could not be updated on the fly.
Solution 3
Non-magnetized nodes were automatically assigned null vector as their force instead of going through calculations. User interaction model was further modified to magnetize selected nodes. Magnetized nodes are given an attractive force towards interactive magnet.

Node Coloring

Challenge 1
Original creator’s highlight function was designed to specifically for highlighting a single node, which meant that there was little to no room to extend the function to allow for highlighting of more than 1 node. Moreover, the highlighting was transient and would also not allow for persistent highlighting of nodes.
Solution 1
Highlighting function was re-implemented to allow for multiple highlights while preserving current functionality. Highlights were assigned to all nodes and made visible as necessary, allowing for coloration and modification of nodes on the fly.
